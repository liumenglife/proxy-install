# proxy-ui 增强设计规格

## 1. 背景与目标

当前项目使用自研 `proxy-ui` 作为 sing-box Clash API 的同源 Web 控制台，浏览器访问 `http://192.168.100.135:9091`，前端请求同源 `/api`，由 proxy-ui 服务端转发到 `http://host.docker.internal:9090`。本阶段继续使用自研 `proxy-ui`，不切换回 MetaCubeXD，也不引入需要用户手动配置后端地址的通用面板。

本次增强目标是把 `proxy-ui` 做成可日常使用的代理控制台：清晰展示当前路由、手动组、自动组、节点延时、可用性和真实可得的速度数据；提供全部测速、按分组测速、定位当前节点、全部展开、全部收起等基础能力；提供受限的 sing-box 内核重启入口；提供符合当前阶段语义的“启动代理/关闭代理”开关。

核心目标：

- 页面去掉 `SING-BOX CLASH API` 字样，避免暴露实现细节给普通使用者。
- 页面自动实时刷新；刷新按钮只表示“立即同步后台状态”，不是页面唯一刷新机制。
- 手动组统一以“手动组”为顶层静态标题，不在每个分组卡片内重复写“一级分组/手动组”。
- 分组展示结构固定为“全部聚合、按机场、按地区”。
- 每个手动组展示可用节点数/总节点数、当前节点、延时、可用性颜色；速度只在当前活跃或当前选择节点有真实数据来源时展示，否则显示 `--`。
- 节点列表展示真实节点名称，不使用截断后无法辨识的别名作为主要文本。
- 自动组只读提示直接展示在页面内，不使用弹窗阻断用户操作。
- UI 必须完成视觉优化，达到现代深色控制台标准，并通过截图验收。

## 2. 非目标与阶段边界

本阶段只增强 `proxy-ui` 与其必要的同源服务端能力，不扩大为完整网关控制系统。

明确非目标：

- 不控制客户端系统代理设置，不修改用户电脑或手机的系统代理、Wi-Fi 代理、浏览器代理配置。
- 不停止 sing-box 容器，不提供“停止容器”或“关闭服务”能力。
- 不在本阶段讨论或实现 TUN 模式下“启动代理/关闭代理”的最终语义；TUN 二阶段再讨论是否代表透明接管、路由规则切换或入站启停。
- 不替换订阅管理流程，不改造 sub-store 的订阅导入、导出和机场解析机制。
- 不把自动组改成可写选择器；自动组仍按 sing-box/urltest 的语义只读展示。
- 不承诺所有节点都有速度数据；没有真实来源时必须显示 `--`，不能用估算值或假数据填充。

当前阶段“关闭代理”不等于关闭容器，也不等于禁用客户端系统代理。它仅表示把代理选择标签切换到 `direct`，让通过当前 proxy-ui 管理路径的默认出站走直连；“启动代理”表示恢复上次代理组，如果没有上次记录则恢复默认 `全部聚合/自动组`。

## 3. UI 信息架构与视觉要求

页面信息架构从上到下分为四层：

1. 顶部控制区：产品名称、后台同步状态、最后刷新时间、启动/关闭代理开关、重启 sing-box 内核按钮、手动同步按钮。
2. 状态卡区：实际路由标签、当前组、当前节点、延时、速度、可用性状态、连接/流量摘要。
3. 操作工具条：全部测速、当前/指定分组测速、定位当前节点、全部展开、全部收起。
4. 分组内容区：手动组和自动组分区展示；手动组按“全部聚合、按机场、按地区”组织；自动组展示只读提示与当前状态。

视觉验收标准：

- 整体风格为现代深色控制台，背景、卡片、边框、状态颜色具备清晰层次，不能是默认浏览器控件堆叠。
- 顶部状态卡需要突出“当前是否代理中、实际路由、当前节点、延时/速度”，用户打开页面 3 秒内能知道当前是否在走代理。
- 分区卡片需要留白合理，标题、摘要、操作按钮、节点列表层级清楚；同类信息不能在多个位置重复造成拥挤。
- 移动端宽度下可用，核心状态、开关、测速、节点选择不横向溢出；节点名称过长时允许单行截断加悬停/展开查看完整文本，但主要列表必须保留真实节点名称。
- 截图验收时不得出现“一级分组/手动组”在每个卡片重复刷屏，不得出现弹窗式只读提示，不得出现大片未对齐按钮或表格拥挤。
- 状态颜色固定语义：深绿色表示可用且延时优秀，浅绿色表示可用且延时良好，黄色表示可用但延时偏高，紫色表示可用但体验差，红色表示 timeout 或接口错误导致不可用，灰色表示无测速数据。

## 4. 数据模型

前端状态由以下数据组成，数据必须来自 sing-box Clash API、proxy-ui 服务端缓存或 control-agent 白名单动作返回，不能伪造业务值。

### 4.1 proxies

`proxies` 表示从 Clash API 获取的代理和代理组集合，至少包含：

- `name`：真实代理或分组名称，用作展示主文本和 API 操作目标。
- `type`：代理类型或分组类型，用于区分 `Selector`、`URLTest`、`Direct`、普通节点等。
- `now`：分组当前选中项。
- `all`：分组内所有候选项。
- `history`：后端已有延时历史，作为 delay cache 的输入来源之一。

### 4.2 selector

`selector` 表示可写手动组。识别规则为 sing-box Clash API 暴露的可切换 selector 分组，并按命名结构归入“全部聚合、按机场、按地区”。每个 selector 需要派生：

- `groupName`：真实分组名，例如 `全部聚合/手动组`、`按机场/<机场名>/手动组`、`按地区/<地区名>/手动组`。
- `displayScope`：展示分区，取值为 `全部聚合`、`按机场`、`按地区`。
- `currentNode`：当前选中节点，即 `now`。
- `nodes`：候选节点真实名称列表，即 `all`。
- `availableCount` 和 `totalCount`：由 delay cache 和节点列表计算。

### 4.3 delay cache

`delay cache` 保存节点和分组的最近测速结果，用于实时刷新期间保持颜色和数值稳定：

- `targetName`：节点或分组名称。
- `delayMs`：成功测速的毫秒数。
- `status`：`excellent`、`good`、`warning`、`poor`、`timeout`、`unknown`。
- `testedAt`：测速完成时间。
- `source`：`history`、`manual-test`、`group-test`、`all-test`。

状态映射：

- `delayMs < 500` 为深绿色可用且延时优秀。
- `500 <= delayMs < 1000` 为浅绿色可用且延时良好。
- `1000 <= delayMs < 2000` 为黄色可用但延时偏高。
- `delayMs >= 2000` 为紫色可用但体验差。
- timeout 或接口错误为红色不可用。
- 无测速数据为灰色未知。

### 4.4 traffic/connections

`traffic/connections` 表示连接和流量摘要。速度字段只在后端 API 能提供当前活跃连接、当前选择节点或总流量速率的真实来源时展示。若无法把速度可靠归因到当前活跃/当前选择节点，节点速度显示 `--`。状态卡可展示整体上下行速率，但不得把整体速率误标为某个节点速度。

### 4.5 group stats

`group stats` 为 UI 派生模型，按每个手动组计算：

- `availableCount`：最近测速结果为深绿色、浅绿色、黄色或紫色的节点数；timeout、接口错误和灰色未知不计入可用节点数。
- `totalCount`：该组候选节点总数。
- `currentNodeDelay`：当前节点延时，优先使用最近测速结果，其次使用 API history。
- `currentNodeSpeed`：当前节点真实速度，无真实来源时为 `--`。
- `availabilityColor`：根据当前节点状态和可用节点比例综合展示，timeout 或接口错误导致节点不可用时优先红色。

## 5. API 设计

### 5.1 proxy-ui 同源 `/api`

前端只访问同源 `/api`，不直接访问 `9090`。proxy-ui 服务端继续把 `/api/*` 转发到 sing-box Clash API。必须保留“不需要填写后端地址、不需要输入密钥”的使用体验。

建议使用的 API 形态：

- `GET /api/proxies`：获取代理、分组、当前选择与历史延时。
- `PUT /api/proxies/:name`：切换 selector 当前节点，body 包含目标节点名称。
- `GET /api/connections`：获取连接和流量信息，用于真实速度展示。
- `GET /api/traffic`：如果当前后端支持流式或轮询流量数据，可用于整体速率卡片；否则 UI 只展示可得数据。
- `GET /api/version` 或等价接口：用于后台连通性和 sing-box 状态展示。

### 5.2 测速调用

测速必须通过同源 `/api` 调用 sing-box Clash API 的 delay 能力，前端不跨域访问 sing-box。调用需要包含目标代理名称、测速 URL 和超时时间。默认测速 URL 使用当前 Clash API 支持的外部探测地址，超时时间由 UI 固定配置，不要求用户输入。

测速结果写入前端 delay cache，并可在本次页面会话内复用。timeout 或接口错误必须写入红色 timeout 状态，并在卡片内以非阻断方式显示错误摘要。

### 5.3 control-agent 白名单接口

新增受限 control-agent 接口仅允许白名单动作，不提供任意命令执行能力。proxy-ui 服务端只能调用明确动作名，不能把用户输入拼接成 shell 命令。

白名单动作：

- `restart-sing-box`：重启 sing-box 内核或容器中的 sing-box 进程，返回动作是否提交成功、执行结果摘要和时间戳。

推荐同源封装：

- `POST /api/control/actions/restart-sing-box`：proxy-ui 服务端调用 control-agent 白名单动作。

接口安全要求：

- control-agent 只接受固定动作名和固定参数结构。
- 响应不得返回敏感环境变量、完整系统日志或可执行命令内容。
- 前端按钮必须有确认态或短暂防重复提交状态，防止连续点击触发多次重启。

## 6. 手动组展示规则与映射规则

手动组顶部只显示一个静态标题“手动组”。标题下按固定顶层顺序展示，任何自动刷新、测速状态更新或重新渲染都不得改变该顺序：

1. `全部聚合`：展示 `全部聚合/手动组`。
2. `按机场`：展示所有匹配 `按机场/<机场名>/手动组` 的 selector。
3. `按地区`：展示所有匹配 `按地区/<地区名>/手动组` 的 selector。
4. `未分类`：仅展示无法归入前三类但仍需保留的 selector 或节点兜底信息。

全局排序规则必须统一用于“自动组代理选择器”下拉框、“手动组代理”分组展示和相关测试断言，不得按接口返回顺序、拼音临时排序或延时结果随意重排顶层分组：

1. 顶层顺序：`全部聚合` -> `按机场` -> `按地区` -> `未分类`。
2. 机场顺序：`猫熊机场` -> `穿墙猫机场` -> `穿山甲机场` -> `瞬云机场`；未知机场排在已知机场之后，并按中文名称排序。
3. 地区顺序：
   - 常用地区：`香港` -> `台湾` -> `澳门` -> `美国` -> `日本` -> `英国` -> `法国` -> `德国`。
   - 亚洲/东南亚：`泰国` -> `菲律宾` -> `马来西亚` -> `印尼` -> `新加坡` -> `越南`。
   - 亚洲/南亚：`巴基斯坦` -> `印度`。
   - 亚洲/中东/西亚：`土耳其` -> `沙特` -> `阿曼` -> `巴林` -> `卡塔尔` -> `伊拉克`。
   - 欧洲：`俄罗斯` -> `乌克兰` -> `荷兰`。
   - 北美洲：`加拿大`。
   - 大洋洲：`澳大利亚`。
   - 南美洲：`巴西`。
   - 未分类：`其他`。
4. 自动组代理选择器下拉顺序：先展示 `全部聚合/自动组`，再按机场顺序展示 `按机场/<机场>/自动组`，再按地区顺序展示 `按地区/<地区>/自动组`，最后展示其他未知自动组，未知自动组按中文名称排序。
5. 手动组分组顺序：先展示 `全部聚合/手动组`，再按机场顺序展示 `按机场/<机场>/手动组`，再按地区层级顺序展示 `按地区/<地区>/手动组`，最后展示 `未分类`/`其他`。
6. 节点列表内部排序：每次自动刷新或手动刷新完成后，按最近一次测速延时稳定排序；有延时节点按真实延时升序排列，`>=2000ms` 紫色节点仍按真实延时参与升序并排在 timeout 前，timeout 或接口错误的红色节点排在有延时节点之后，无测速数据的灰色未知节点排在最后；相同状态或相同延时保留订阅导出顺序作为 tie-breaker，避免跳动。机场、地区和顶层分组顺序仍按固定顺序，不受测速影响。

每个手动组卡片必须展示：

- 真实分组名称或清晰的人类可读标题。
- 可用节点数/总节点数。
- 当前节点真实名称。
- 当前节点延时和颜色。
- 当前节点速度；无真实数据来源时显示 `--`。
- 节点列表，列表项使用真实节点名称，并展示延时颜色、当前选中态和可用性。

地区映射优先使用分组名中的 `<地区名>`；如果只有节点名可用，则根据节点真实名称中的地区关键词映射。无法可靠识别地区的节点归入“未分类”。未分类不是错误状态，必须正常展示和测速。

机场映射优先使用 `按机场/<机场名>/手动组` 中的 `<机场名>`；如果缺少该结构，不从节点名猜测机场，避免错误归类。

## 7. 测速功能规则

测速入口包含：

- 全部测速：对当前可见或全部已知节点执行测速，更新 delay cache 和所有组统计。
- 按分组测速：只对指定手动组内候选节点执行测速，更新该组和相关节点状态。
- 定位当前节点：滚动并展开当前实际路由涉及的组和节点，突出当前节点。
- 全部展开：展开所有手动组和自动组卡片。
- 全部收起：收起所有可折叠卡片，但顶部状态卡保持可见。

测速状态规则：

- `<500ms`：深绿色，可用且延时优秀。
- `500ms <= delay < 1000ms`：浅绿色，可用且延时良好。
- `1000ms <= delay < 2000ms`：黄色，可用但延时偏高。
- `>=2000ms`：紫色，可用但体验差。
- `timeout` 或接口错误：红色不可用，不计入可用节点数。
- 没有测速数据：灰色未知，不计入可用节点数。

并发限制：

- 全部测速必须限制并发，避免同时向大量节点发起请求导致浏览器、proxy-ui 服务端或 sing-box API 被打满。
- 默认并发上限为 6；按组测速也使用同一上限。
- 同一节点在一个测速批次内只测速一次，多个分组复用同一结果。
- 批次执行期间需要显示进度，例如“已完成 12/80”，并允许用户继续查看页面。

超时规则：

- 单个节点测速超时时间为 5000ms。
- 超时或接口错误节点写入 delay cache 的 `timeout` 状态。
- 批次中部分失败不影响其他节点继续测速。

## 8. 开关与重启规则

### 8.1 启动代理/关闭代理

UI 文案必须使用“启动代理”和“关闭代理”，不能显示为“切 direct”“停容器”“停 TUN”。

当前阶段底层实现：

- 关闭代理：把代理选择标签切换到 `direct`。
- 启动代理：恢复上次代理组；如果本地没有上次记录或后台目标不存在，则恢复默认 `全部聚合/自动组`。
- UI 需要记录关闭前的上次代理组和节点，记录只用于恢复选择，不作为安全配置或持久系统状态。

状态判定：

- 当前实际路由为 `direct` 时，显示“代理已关闭”。
- 当前实际路由为非 `direct` 且有可识别代理组/节点时，显示“代理已启动”。
- 后台状态无法读取时，开关进入不可用态，并显示同步失败原因摘要。

### 8.2 重启 sing-box 内核

“重启 sing-box 内核”按钮通过 control-agent 白名单动作实现。按钮规则：

- 点击后进入确认态，确认后提交 `restart-sing-box` 动作。
- 动作提交后按钮进入处理中状态，避免重复提交。
- 成功后等待后台恢复，再重新同步 `/api/proxies`、连接信息和版本/健康状态。
- 失败时在页面内展示错误摘要，不弹出阻断式弹窗。

## 9. 实时刷新规则

页面必须自动实时刷新，手动刷新按钮只作为“立即同步后台状态”。

刷新频率：

- 代理组、当前选择、延时历史：每 3 秒轮询一次。
- 连接和整体流量摘要：每 1 秒轮询一次；如果后端只支持较低频率，则降级到 3 秒。
- 重启 sing-box 后恢复检测：前 15 秒每 1 秒检测一次，之后恢复常规频率。

刷新行为：

- 自动刷新不得覆盖用户正在展开/收起的卡片状态。
- 自动刷新不得把用户滚动位置强制拉回顶部。
- 自动刷新不得中断正在进行的测速批次。
- 当用户正在打开某个分组查看节点时，只按本 spec 的延时排序规则稳定更新该分组内节点顺序、数值和状态，不重建无关 DOM 导致焦点丢失。
- 手动刷新按钮只立即触发一次后台同步，并更新“最后同步时间”和错误状态。

数据冲突处理：

- 如果用户刚发起切换节点，短时间内后台仍返回旧 `now`，UI 可显示“同步中”，但最终必须以后台确认状态为准。
- 如果后台返回的分组或节点不存在于本地展开状态，保留其他展开状态，移除无效项。

## 10. 测试与验收

实现完成前必须经过用户确认本 spec，再确认 implementation plan。

验收闭环包括：

- UT：覆盖分组分类、机场排序、地区排序、自动组代理选择器下拉排序、手动组分组排序、节点列表按刷新后延时升序稳定排序、同延时/同状态保留订阅导出顺序、手动组统计、深绿色/浅绿色/黄色/紫色/红色/灰色延时颜色、速度 `--` 规则、启动/关闭代理目标选择、control-agent 白名单请求构造。
- proxy-ui 服务端测试：覆盖 `/api` 同源转发、测速代理调用、control-agent 白名单接口不会接受任意命令。
- Playwright CLI：覆盖页面加载、自动刷新、手动同步、全部测速、按组测速、定位当前节点、全部展开、全部收起、启动/关闭代理、重启按钮确认态，并断言自动组代理选择器下拉顺序、手动组分组顺序、节点列表刷新后延时升序稳定排序和延时颜色分级符合本 spec。
- QA 出口 IP：在代理启动和关闭后分别检查出口行为是否符合当前阶段定义；关闭代理只验证 proxy-ui 管理路径切到 `direct`，不验证客户端系统代理被关闭。
- 视觉截图：桌面和移动端各至少一张，检查现代深色控制台、顶部状态卡、分区卡片、清晰层级、无拥挤重复文本、无弹窗式只读提示。
- Code Review：只读审核功能正确性、安全边界和错误处理。
- QA 审核：只读审核交互完整性、移动端可用性、重启/测速失败场景。
- Spec 满足度审查：逐条对照本 spec 的目标、非目标、UI、API、测试验收项，结论必须明确“通过”或“不通过”。

关键验收场景：

- 页面顶部不再出现 `SING-BOX CLASH API`。
- 打开页面后无需点击刷新，状态会自动更新。
- 手动组顶部只出现一次“手动组”，每个卡片不重复“一级分组/手动组”。
- 自动组代理选择器下拉顺序严格为 `全部聚合/自动组`、按机场顺序、按地区顺序、未知自动组中文名称排序。
- 手动组分组顺序严格为 `全部聚合/手动组`、按机场顺序、按地区层级顺序、`未分类`/`其他`。
- 地区按指定层级和地区顺序展示，无法识别的节点进入“未分类”或“其他”。
- 节点列表每次刷新后按最近一次测速延时升序稳定排序：有延时节点升序，timeout/接口错误红色节点随后，无测速灰色节点最后；相同状态或相同延时保留订阅导出顺序作为 tie-breaker，机场/地区分组顺序不受测速影响。
- 节点列表展示真实节点名称。
- 自动组只读提示在页面内展示，不弹窗。
- 当前活跃/当前选择节点无真实速度来源时显示 `--`。
- 延时颜色和可用性严格符合以下规则：
- `<500ms` 深绿色可用且延时优秀。
- `500ms <= delay < 1000ms` 浅绿色可用且延时良好。
- `1000ms <= delay < 2000ms` 黄色可用但延时偏高。
- `>=2000ms` 紫色可用但体验差。
- `timeout`/接口错误红色不可用。
- 无测速数据灰色未知且不计入可用节点数。
- 重启 sing-box 内核只能通过 control-agent 白名单动作触发。

## 11. 风险与限制

- sing-box Clash API 暴露字段可能与 MetaCubeXD 使用的 mihomo 字段不完全一致，实现时需要以当前 API 实测响应为准，但不得改变本 spec 的用户可见语义。
- 节点数量较多时，全部测速可能耗时较长；并发限制会牺牲完成速度以换取稳定性。
- 速度数据归因存在天然限制；无法可靠归因到当前节点时必须显示 `--`，这会让部分卡片看起来信息较少，但比展示假数据更正确。
- “启动代理/关闭代理”在当前阶段只是 selector/direct 切换，不代表系统级代理状态；UI 必须用说明文案降低误解。
- 重启 sing-box 期间 API 会短暂不可用，页面需要展示恢复中状态，避免误判为永久失败。
- 地区识别依赖分组名和节点名关键词，无法覆盖所有命名风格；未分类展示是预期兜底。
- control-agent 增加了新的控制面，必须保持白名单、固定动作和最小返回信息，避免变成远程命令执行入口。
